# China-Stock-Analyst 独立审查报告

**审查日期**: 2026-04-03  
**审查方式**: README 基线阅读 + 核心脚本静态审查 + Agent/文档一致性交叉检查 + 测试执行 + 存量报告质量门禁验证  
**说明**: 本报告参考了既有两份审查文档，但所有结论均以当前仓库实际代码与验证结果为准，不修改既有报告、不修改源代码。

---

## 一、总体结论

**综合判断**: 项目已经具备较强的工程化意识，尤其在 Team-First 路由、数据真实性门禁、身份校验、东财接口封装、回归测试方面表现较好；但当前仍存在**架构集中、文档与实现口径漂移、门禁链路局部误伤、报告后置检查覆盖不足**等问题。

**当前定位建议**:

- 适合作为 **A 股短线分析的决策支持工具**
- 不适合作为 **自动化交易执行系统**
- 如果用于实际交易决策，仍应保留人工复核，尤其是价格锚点、财报期、东财复核状态和报告时间锚点

**我的独立评分**:

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构与可维护性 | 7.0/10 | 路由清晰，但 `generate_report.py` 职责过重 |
| 业务安全性 | 6.8/10 | 门禁意识强，但仍有误降级和新鲜度误判风险 |
| 测试与验证 | 7.8/10 | 单测覆盖广，后置门禁脚本覆盖不足 |
| 文档一致性 | 6.5/10 | README/Agent/Schema/实现存在多处口径偏差 |
| 综合 | **7.0/10** | 可用，但不建议无人工复核直接驱动交易 |

## 三、关键问题清单

### P0：高优先级问题

#### 1. Agent 命名与 Schema 规范仍未闭合

这是我认为当前最明确、最持续存在的问题。

**现状**:

- Agent 文件 `name` 使用 `stock-industry-researcher`、`stock-event-hunter`
- Schema 标准要求 `agent` 字段与 `name` 对应
- 但实际代码生成的 `agent` 值却是 `expert_industry_researcher`、`expert_event_hunter`
- 身份校验返回的 `agent` 又是 `expert_identifier_agent`

**风险**:

- 聚合层、校验层、Prompt 规范层不是一个命名体系
- 现在之所以能工作，是因为代码内部常量和测试在“兼容当前实现”
- 后续一旦真正接入外部 Agent 输出、做 schema 校验、或新增专家，极易出现隐藏性断链

**结论**:

- 这是“规范未收口”问题，而不是单纯文档问题
- 该问题优先级高，因为它会影响后续扩展与自动校验能力

---

#### 2. 采集质量门禁对东财质量信息存在默认失败路径

`parse_search_results_to_report()` 会执行 `_run_collection_quality_gate()`，而该门禁依赖：

- `data_quality_summary`
- `field_conflict_summary`
- 或 `eastmoney_router.critical_gate`

但在纯 Web Search 路径下，这些结构并不会天然填充到 `report` 中。

**结果**:

- 只要没有补充到东财质量信息，门禁就可能走到“缺少东财结构化质量信息”
- 然后整体 `confidence_level` 被降为 `低`

**风险**:

- 这不是“严格风控”，而是“门禁输入未对齐导致的系统性误降级”
- 用户会把这种降级理解为数据真的低质，但实际上可能只是链路集成不完整

**结论**:

- 这是当前业务可信度里最值得警惕的问题之一
- 它会直接影响报告标签与置信度解释

---

#### 3. 财务日期缺失时直接回退为当天，存在“伪新鲜度”

`_parse_financial()` 在提取不到财报期时，直接将 `as_of` 设置为当天日期。

**风险**:

- 历史财报文本可能被包装成“今天的有效财务快照”
- 用户会误判数据时效
- 后续的真实性与新鲜度判断也会被污染

**结论**:

- 这是典型的“兜底值污染业务语义”
- 在金融场景里，这类问题比普通字段缺失更危险

---

### P1：重要问题

#### 4. `generate_report.py` 仍然过于集中

当前文件同时承担：

- 搜索结果解析
- 多源审计
- 采集质量门禁
- 舆情治理
- 专家输出构造
- 身份校验
- 主管仲裁
- Markdown 渲染

**问题本质**:

- 这不是“行数多”本身的问题
- 而是一个文件承担了“规则引擎 + 业务编排 + 输出渲染”三类职责

**风险**:

- 未来改一个降级规则，可能影响报告渲染或专家仲裁
- 测试即便通过，也会越来越依赖大量回归用例来保护局部改动

---

#### 5. README 描述的专家链条比当前真实实现更完整

文档层描述了完整的专家固定链：

- fundamental
- technical
- quant
- risk
- macro
- industry
- event
- identity

但当前 `parse_search_results_to_report()` 中显式实际构造的专家输出，主要集中在：

- industry
- event
- identity
- supervisor

Markdown 渲染虽然兼容 fundamental 等输出结构，但主流程里并未看到同等完整的生成闭环。

**风险**:

- 用户会以为自己拿到的是“全专家并行结果”
- 实际上部分专家更像“架构目标”或“渲染兼容位”

**结论**:

- 这不是功能完全缺失，而是“文档承诺强于当前落地”

---

#### 6. `agent_team` 模式会强制落成 `stock_pool` 报告模式

`format_obsidian_markdown_report()` 中，只要 `analysis_mode == "agent_team"`，就直接按 `stock_pool` 输出。

**风险**:

- 单标的 Team 模式报告也可能被当成股票池模式处理
- 这与 README 中“单标的自动 single_stock、多标的自动 stock_pool”的表述不完全一致

**结论**:

- 这是实现与产品语义之间的小型错位
- 风险不如 P0 高，但会影响输出一致性和用户预期

---

#### 7. 东财选股的放宽重试会额外消耗配额

`eastmoney_stock_screen()` 在空结果时，会继续做：

- `price_only`
- `keyword_only`

两轮放宽查询。

**风险**:

- 每次重试都可能再次消耗日配额
- 单次用户请求的真实额度成本并不透明

**结论**:

- 在 50 次/日的限制下，这属于需要明确治理的资源问题

---

#### 8. 报告质量门禁脚本虽可用，但检查范围仍偏窄

当前 `report_quality_gate.py` 主要检查：

- 候选表“待确认”却写死收盘价
- 推荐段收盘价与候选表偏差超过 2%
- 推荐段缺少时间锚点

**不足**:

- 没校验止损价逻辑
- 没校验标签与综合分一致性
- 没校验风险等级与推荐标签冲突

**而且实测结果显示问题真实存在**:

- 运行 `python scripts/run_report_quality_checks.py`
- 共检查 3 份报告
- 其中 **1 份未通过**
- 具体包含：
  - 3 个缺少时间锚点问题
  - 1 个“候选表待确认但推荐段写入确定收盘价”问题

**结论**:

- 当前质量门禁不是摆设，能查出真实问题
- 但它仍然偏“最低可用”，未形成全面质量闭环

---

### P2：中优先级问题

#### 9. 报告后置工具链测试不足

虽然 115 项测试全部通过，但测试重心主要在：

- 路由
- 解析
- 审计
- 身份校验
- 东财解析

而对以下脚本的直接测试明显不足：

- `report_quality_gate.py`
- `run_report_quality_checks.py`
- CLI 退出码路径
- 网络失败与重试分支

这意味着“核心函数层很稳”，但“工具链与边缘脚本层没有同等强度的回归保护”。

---

#### 10. 路由缓存与计数文件更偏单进程设计

当前存在多个本地 JSON 持久化文件：

- 路由缓存
- 路由日志
- 东财日配额计数

代码使用了线程锁，但没有看到跨进程级别的原子写保护。

**风险**:

- 单用户单进程下问题不大
- 如果未来进入多进程、并发调用或异常中断场景，存在文件损坏与状态竞争风险

---

#### 11. 部分 helper 函数属于“低层关键逻辑，高层间接保护”

例如来源分类、时间标准化、时间提取等 helper，会影响：

- 数据真实性审计
- 来源类别充分性判断
- 时间冲突判断

目前这类 helper 更多是被高层测试“间接覆盖”，而不是被独立、精确地回归保护。

这类问题不会马上爆雷，但会提升后续维护时的回归不确定性。

   - 当前已是 115 项，并非旧值

2. **API Key 文档明显不一致**  
   - 当前基本已经统一为“三种变量兼容，推荐 `EASTMONEY_APIKEY`”

3. **质量门禁只依赖单一章节标题**  
   - 当前已改成多个标题模式匹配，说明问题已部分处理

## 六、建议优先级

### 第一阶段：先修可信度闭环

1. 统一 Agent 命名体系  
   - `name / agent / schema / 常量 / 测试` 必须完全收口为一套标准

2. 修正采集质量门禁输入契约  
   - 明确纯 Web Search、东财复核成功、东财不可用三种状态
   - 避免把“未接入”判成“校验失败”

3. 去除财务日期的“当天兜底”语义污染  
   - 缺失就应显式缺失，而不是伪造当天

### 第二阶段：再修实现与文档漂移

4. 把 README / SKILL 中的“全专家闭环”描述，与当前真实生成链路对齐
5. 修正 `agent_team` 与 `single_stock/stock_pool` 的模式映射语义
6. 明确单次东财请求的真实额度成本与放宽策略

### 第三阶段：补强质量工具链

7. 给 `report_quality_gate.py` 和 `run_report_quality_checks.py` 补专门测试
8. 扩大质量门禁规则：止损位、标签-分数一致性、风险-建议冲突
9. 拆分 `generate_report.py`，至少分离“解析/门禁/仲裁/渲染”

## 七、最终判断

这是一个**已经具备明显产品雏形和工程意识**的项目，不是“玩具脚本”；但它离“可放心长期维护的稳定金融分析系统”还有关键几步。

如果让我给一句话结论：

> **项目已经可用于研究辅助与决策参考，但仍不宜把当前输出视为可直接执行的交易指令。**

核心原因不是“功能不够多”，而是：

- 规范还没完全收口
- 部分门禁存在误伤或语义污染
- 文档承诺与真实落地还没有完全对齐
